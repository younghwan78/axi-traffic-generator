SoC Multimedia DMA Traffic Generator 구현 작업지시서 (바이브 코딩용)
1. 프로젝트 개요 (Project Overview)
본 프로젝트는 SoC 멀티미디어 IP(Common DMA v16.0)의 메모리 접근 트래픽 패턴(Address, Size, Burst)을 생성하는 Python 기반 시뮬레이터 개발입니다. Cycle-accurate한 AXI/ACE-lite 프로토콜 시뮬레이션이 아닌, 실제 동작과 유사한 버스 트랜잭션 시퀀스(Address Pattern)를 NoC(Network on Chip) 분석용 포맷으로 출력하는 것이 목적입니다.

2. 핵심 설계 철학 (Core Architecture)
Generator Pattern: 파이썬의 yield를 사용하여 수백만 건의 트랜잭션을 메모리 오버헤드 없이 생성하고 Interleaving 처리.
Separation of Concerns: 정적 하드웨어 스펙(DMA_IP_Spec.yaml)과 동적 유스케이스(Scenario.yaml)를 완전히 분리.
Virtual Tick & Scoreboard: 나노초 단위의 절대 시간이 아닌, Clock Domain 기반의 가상 틱(Virtual Tick)과 Producer-Consumer 종속성을 Scoreboard로 제어.
Mock SMMU: 선형적인 IOVA(I/O Virtual Address)를 물리 주소(PA)로 변환하며, 4KB 페이지 단위의 파편화(Fragmentation) 및 SMMU PTW(Page Table Walk) Read 트래픽을 모델링.

3. 모듈별 구현 요구사항 (Module Specifications)
Module 1: Configuration Parser (config_parser.py)
PyYAML을 사용하여 두 개의 YAML 파일을 파싱합니다.
DMA_IP_Spec.yaml: IP의 물리적 속성 (BusByte, PPC 등). (Inline Dictionary 구조)
Scenario.yaml: 시나리오별 Clock Domain 주파수, 해상도, 포맷(YUV/RGB), DMA 간 Dependency 정의.

Module 2: Mock SMMU & PA Translator (smmu_model.py)
기능: 가상 주소(IOVA)를 입력받아 4KB 단위 물리 페이지(PA)로 매핑.
Fragmentation Ratio 지원: 시나리오에서 설정한 비율(예: CMA 30%, SG 70%)에 따라 물리 메모리의 연속성 여부를 결정.
PTW (Page Table Walk) Injection: 새로운 4KB 페이지에 최초 접근 시(TLB Miss), SMMU가 Page Table을 읽어오는 64Byte Read 트랜잭션을 트래픽 로그에 삽입할 수 있도록 제너레이터에 신호(또는 트랜잭션 객체)를 반환.

Module 3: DMA Address Generator (dma_generator.py)
데이터 클래스 (Transaction): ch_id, ip_name, rw('W'/'R'), iova, pa, size, burst('seq').
64B Boundary Chopping: 모든 트랜잭션은 단일 크기가 64Byte를 넘을 수 없으며, 주소가 64Byte 정렬 경계를 넘어가면 무조건 2개 이상의 트랜잭션으로 분할(yield).
Access Patterns: * raster-order: Line 단위 선형 접근 (Stride 고려).
Z-order: Fixed size Tile (64x32, 64x64) 단위 접근.
Multi-Plane 지원: YUV420 2-Plane (NV12) 등 멀티 플레인 포맷의 경우 Y와 UV의 제너레이터를 병렬로 생성.

Module 4: Virtual Tick Scheduler & Scoreboard (scheduler.py)
SystemScoreboard: IP가 Write를 수행한 Line(또는 Tile) 번호를 기록.
Dependency Check: Consumer IP는 Wait_For로 지정된 Producer IP가 해당 영역 처리를 완료할 때까지 Stall (트랜잭션 멈춤).
Virtual Tick Arbiter: Scenario.yaml의 Clock(Freq)과 IP 스펙의 PPC, BusByte를 바탕으로 대역폭 비율을 계산하여 Tick 당 동작할 IP를 라운드 로빈 스케줄링.

Module 5: NoC Traffic Exporter (exporter.py)
생성된 트랜잭션 리스트를 NoC 담당자가 요구하는 특정 텍스트 포맷으로 출력.
출력 포맷 규격: id={id} port={ip_name} type={type} address={pa} bytes={size} burst=seq
type: Write일 경우 WriteNoSnoop, Read일 경우 ReadNoSnoop으로 매핑.
address: SMMU를 통과한 최종 물리 주소 (Hex string).

6. 수학적 모델링 제약 (Mathematical Modeling Constraints) 및 구현 옵션AI 코딩 어시스턴트는
다음의 하드웨어적 배경지식(Why)과 구현 옵션(Options)을 숙지하여 제너레이터(Generator) 알고리즘을 설계해야 합니다.
6.1. 64B Boundary Chopping (64바이트 경계 분할)왜 이 제약이 필요한가? 
(Why it's needed):AXI 버스 프로토콜과 SoC 내부의 메모리 컨트롤러(DRAM Controller)는 
일반적으로 Cache Line 크기인 64Byte(또는 128Byte) 단위로 최적화되어 있습니다. 
단일 Burst 트랜잭션이 64Byte 정렬 경계(예: 0x...40, 0x...80, 0x...C0)를 걸쳐서(Crossing) 넘어가게 되면 
하드웨어 캐시 라인 미스나 AXI Protocol Error가 발생할 수 있습니다. 
따라서 DMA는 이를 방지하기 위해 하드웨어적으로 트랜잭션을 쪼개서(Chopping) 전송하도록 설계되어 있습니다.

구현 옵션 및 가이드 (Options & Guide):
Option A (Strict Alignment): 시작 주소가 비정렬(Unaligned) 상태일 경우, 첫 번째 트랜잭션은 다음 64B 경계까지만 쪼개서 보냅니다. (예: Addr=0x1030, Size=64B 요청 시 $\rightarrow$ [0x1030, 16B] 와 [0x1040, 48B] 로 분할). 
본 프로젝트에서는 이 방식을 기본으로 채택합니다.

Option B (Bus Width & BL 기반 분할): IP 스펙의 BusSizeBytes(16B 또는 32B)를 기준으로 최대 Burst Length(Max BL)를 산출하여 쪼갭니다. 
32B 버스일 경우 Max BL=2, 16B 버스일 경우 Max BL=4로 계산하여 64B를 맞추는 로직을 제너레이터에 포함해야 합니다.

6.2. Z-order Access / Tiled Memory (타일 형태 메모리 접근)왜 이 제약이 필요한가? 
(Why it's needed):멀티미디어(특히 Camera ISP, Video Codec)에서는 2D 블록 단위(예: 8x8, 64x32) 연산이 빈번합니다. 
이를 일반적인 선형(Raster) 구조로 메모리에 올리면, Y축으로 1픽셀 내려갈 때마다 메모리 주소가 Width(Stride)만큼 크게 건너뛰게 되어 DRAM의 Page Miss(Row buffer conflict)가 극심하게 발생합니다. 
이를 막기 위해 2D 공간의 지역성(Spatial Locality)을 메모리 주소 상의 연속성으로 변환하는 Z-order 또는 Tiled 포맷을 사용합니다.

구현 옵션 및 가이드 (Options & Guide):
Option A (Macro-Tile 선형 배치): 64x32 크기의 타일 자체가 하나의 큰 1D 배열처럼 메모리에 연속적으로 Packing되어 있고, 
타일 간의 순서만 Raster로 스캔하는 방식입니다. 
(계산이 단순하여 시뮬레이터 속도가 빠름)

Option B (True Morton Code): X좌표와 Y좌표의 비트를 교차(Interleaving)시켜 실제 수학적인 Z-order 곡선을 그리는 방식입니다. 
(비트 연산이 들어가며 가장 엄밀한 물리 주소 패턴 도출 가능)
본 프로젝트의 요구사항: 시나리오 파라미터로 Tile_Width와 Tile_Height를 입력받아 Option A 방식으로 타일의 시작 주소를 계산하되, 
타일 내부의 데이터를 읽을 때 64B Chopping 제약을 동일하게 적용하는 제너레이터를 구현해야 합니다.

6.3. SMMU Fragmentation (메모리 파편화 및 페이지 테이블 워크)왜 이 제약이 필요한가? 
(Why it's needed):SW(안드로이드 OS)가 할당한 버퍼는 가상 주소(IOVA) 상에서는 연속적이지만, 실제 물리 주소(PA)는 4KB 단위로 흩어져(Scattered) 있습니다. 
NoC 성능 분석 시 이 파편화로 인한 병목과 SMMU가 페이지 매핑 정보를 읽어오기 위해 발생하는 추가적인 읽기 트래픽(PTW: Page Table Walk)이 시스템 대역폭을 갉아먹는 주요 원인이 됩니다.

구현 옵션 및 가이드 (Options & Guide):Option A (Random Allocation): 4KB 범위를 넘어설 때마다 무조건 임의의 물리 주소를 매핑합니다. 
(Worst-case 시나리오 분석에 적합)

Option B (CMA + SG 혼합 모델): OS의 특성을 반영하여 설정된 비율(예: 30%)만큼은 연속된 물리 주소(CMA)를 부여하고, 나머지는 파편화시킵니다.

본 프로젝트의 요구사항: Option B를 구현하며, 새로운 4KB 페이지에 진입할 때(TLB Miss 발생 시), 실제 데이터 트랜잭션이 발생하기 직전에 SMMU가 Page Table을 읽어오는 64Byte Read 트랜잭션을 로그에 우선적으로 yield하는 로직을 반드시 포함해야 합니다.
SMMU는 선택 사항으로 default로는 고려하지 않고 option이 있을때만 이를 고려해서 traffic을 생성합니다.

7. 전체 핵심 구현 기능 명세 (Core Features Checklist)
본 시뮬레이터는 다음의 핵심 기능들을 빠짐없이 구현해야 합니다.

7.1. Configuration & Parsing (설정 및 파싱 엔진)
[ ] 정적 하드웨어 스펙 파서 (DMA_IP_Spec.yaml): IP별 물리적 버스 폭(BusByte), 처리량(PPC), 최대 지원 Plane 수, SBWC 지원 여부 등을 로드하는 기능.

[ ] 동적 시나리오 파서 (Scenario.yaml): 유스케이스별 동작 주파수(Clock Domain), 이미지 포맷(YUV/RGB), 해상도, 타일 크기, IP 간 의존성(Dependency)을 로드하는 기능.

[ ] Sanity Check (정합성 검증): 시나리오에서 요구한 동작(예: SBWC 압축 켜기)이 해당 IP의 정적 스펙에서 지원하지 않는 경우, 시뮬레이션 시작 전에 에러를 출력하고 중단하는 기능.

7.2. Core Traffic Generation (트래픽 생성 코어)
[ ] 64B Boundary Chopping 알고리즘: 단일 트랜잭션이 64Byte를 넘지 않으며, 시작 주소가 비정렬(Unaligned) 상태일 때 64Byte 정렬 경계에서 정확히 트랜잭션을 분할하는 기능.

[ ] Image Format Descriptor: YUV444/422/420, RGB 등 포맷에 따른 Bytes Per Pixel(BPP), Stride, Plane별 해상도(Sub-sampling)를 자동으로 계산하는 기능.

[ ] Multi-Plane Interleaving: YUV 2-Plane, 3-Plane 처리 시 각 Plane의 트래픽을 순차적으로 다 보내는 것이 아니라, DMA 내부 FIFO 비율에 맞춰 교차로(Interleaved) 발생시키는 기능.

[ ] Access Pattern Generator:

Raster-order: Stride를 반영한 Line-by-Line 선형 주소 생성.

Z-order: 지정된 Tile Size(예: 64x32) 단위로 매크로 블록 주소를 스캐닝.

Random 1D/2D: 시나리오에 따라 임의의 위치(Offset)에서 트래픽을 발생.

[ ] SBWC (압축) 트래픽 모델링: (옵션/고급) SBWC 활성화 시, Header 영역과 Payload 영역을 분리하여 읽거나/쓰는 패턴을 생성하며, 32B/64B/128B Alignment 제약을 준수.

7.3. Memory & Address Translation (SMMU 모델 및 주소 변환)
[ ] IOVA to PA 변환기: 제너레이터가 만든 선형 가상 주소(IOVA)를 4KB Page 단위의 물리 주소(PA)로 변환하는 Mock SMMU 테이블 관리.

[ ] Fragmentation (파편화) 모델링: 시나리오의 설정(예: CMA 30%)에 따라 물리 메모리를 연속적으로 할당하거나 파편화(Scatter-Gather)시켜 할당하는 정책 엔진.

[ ] PTW (Page Table Walk) 트래픽 주입: 새로운 4KB 페이지(TLB Miss)에 최초 접근할 때마다, 실제 데이터 접근 직전에 64Byte Read (PTW 트래픽)를 로그에 강제로 추가하는 기능.

7.4. Scheduling & Dependency (가상 시간 및 의존성 관리)
[ ] Virtual Tick 기반 Arbiter: 동작 주파수(Clock)와 PPC 값을 기반으로 가상 사이클을 진행시키며, 여러 DMA의 대역폭 점유율에 따라 Round-Robin 또는 가중치(Qurgent 등) 스케줄링.

[ ] Scoreboard 기반 종속성 관리: * Producer: 데이터 처리(Write)가 완료된 영역(Line 번호 또는 Tile 번호)을 Scoreboard에 업데이트.

Consumer: 자신이 읽어야 할 영역이 Scoreboard에 기록될 때까지 대기(Stall/Yield None) 상태를 유지.

[ ] Margin & Granularity 컨트롤: Consumer가 Producer를 얼마나 바짝 쫓아갈지(예: "10 Line 뒤에서 쫓아감" 또는 "1 Frame 다 쓰면 시작함")를 시나리오에서 설정하고 제어하는 기능.

7.5. Output & Logging (출력 포맷터)
[ ] NoC Traffic Exporter: 생성된 모든 트랜잭션 객체를 NoC 검증팀이 요구하는 규격(id=... port=... type=... address=... bytes=... burst=seq)의 순차적 Text/CSV 파일로 출력하는 기능.

[ ] Read/Write Type 변환 매핑: 내부적인 'R', 'W' 상태를 NoC 포맷에 맞게 ReadNoSnoop, WriteNoSnoop 문자열로 변환.

7.6. (선택적 고급 기능) QoS 및 버퍼 오버플로우 검증
[ ] Qurgent 및 VOTF 반영: 특정 DMA가 마감 시간(Deadline)에 쫓길 때 가상 스케줄러에서 우선순위를 일시적으로 높여주는 로직.

[ ] 내부 Data Fifo 검증: Producer/Consumer 간 속도 차이로 인해 IP 내부의 DataFifoSize를 초과하여 데이터가 쌓이거나 고갈(Underflow)되지 않는지 가상 틱 기반으로 모니터링하는 기능.

8. 미시적 동작 패턴 및 파이프라인 제어 (Micro-Behavioral Profiles)
단순한 주기적 트래픽 생성을 넘어, 실제 멀티미디어 IP의 파이프라인 특성(Stall, Accumulation)을 정확히 모사하기 위해 제너레이터에 동작 프로파일(Behavior Policy) 속성을 부여하고 Strategy Pattern으로 구현해야 합니다. 새로운 동작 패턴이 추가되거나 변경될 수 있도록 확장성을 고려해야 합니다.

8.1. YAML 속성 확장 (Profile 정의)
Scenario.yaml의 Task 정의 부분에 Behavior_Profile 속성을 추가하여 DMA의 동작 방식을 구체화합니다.

YAML
Tasks:
  # 1. Image WDMA (MO 단위 Burst 동작)
  - TaskName: "CAM_FE_Write"
    IP_Name: "CAM_ISP_WR_0"
    Behavior_Profile: 
      Type: "Eager_MO_Burst"       # 데이터가 준비되면 MO 제한까지 즉시 트래픽 생성
      Pipeline_Group: "CAM_FE_PIPE" # 동일 파이프라인 그룹 지정 (Backpressure 공유)

  # 2. Image RDMA (WDMA Stall에 종속)
  - TaskName: "CAM_FE_Read"
    IP_Name: "CAM_ISP_RD_0"
    Behavior_Profile:
      Type: "Eager_MO_Burst"
      Pipeline_Group: "CAM_FE_PIPE"
      Backpressure_Source: "CAM_FE_Write" # WDMA가 Stall 걸리면 이 RDMA도 강제 Stall

  # 3. Stat Data DMA (특정 블록 누적 후 트래픽 발생)
  - TaskName: "CAM_AE_Stat"
    IP_Name: "CAM_STAT_WR_0"
    Behavior_Profile:
      Type: "Accumulate_and_Flush" # 특정 단위가 모였을 때만 트래픽 발생
      Trigger_Unit: "Block"        # Block, Line, Frame 등
      Block_Size: [64, 64]         # 64x64 픽셀이 처리될 때마다
      Flush_Bytes: 256             # 256 Bytes의 Stat 데이터를 한 번에 전송
8.2. Python 구현 요구사항 (Behavioral Strategy)
AI는 DmaAgent 내부에 트래픽 생성 방식을 결정하는 BehaviorStrategy 클래스를 상속받아 다음 두 가지 구체화된 전략을 구현해야 합니다.

[ ] Eager_MO_Strategy (Image Read/Write용):

스케줄러가 Tick을 진행할 때, PPC (Pixels Per Clock) 값만큼 가상 내부 버퍼에 데이터를 적립(Accumulate)합니다.

내부 버퍼에 모인 데이터가 Bus Size(예: 32B)를 넘어가면, IP 스펙에 정의된 required_MO (Multiple Outstanding) 허용치 한도 내에서 가능한 한 번에 여러 개의 트래픽을 연속으로 yield 합니다.

Backpressure 로직: Backpressure_Source로 지정된 타겟 DMA(예: WDMA)가 현재 버스 병목으로 인해 데이터를 내보내지 못하고 내부 버퍼가 꽉 찬 상태(Stall)라면, 자신(RDMA)의 트래픽 생성도 즉시 중단(Yield None)합니다.

[ ] Accumulate_and_Flush_Strategy (Stat Data용):

메인 이미지 파이프라인(예: CAM_FE_PIPE)의 진행률(처리된 픽셀 수 또는 라인 수)을 Scoreboard를 통해 모니터링합니다.

모니터링 중인 픽셀 수가 Block_Size(예: 64x64 = 4096 픽셀) 배수에 도달할 때마다 트리거가 발동됩니다.

트리거 발동 시, Flush_Bytes 크기만큼의 트래픽(예: 256Byte면 64B 트랜잭션 4개)을 일시적으로 Burst 형태로 버스에 쏟아냅니다. 조건이 만족되지 않은 평소(Tick)에는 어떠한 트래픽도 발생시키지 않습니다 (Silent).

이 확장의 핵심 의도
이 지시서를 추가함으로써 AI 코딩 어시스턴트는 단순히 for문을 돌며 주소를 찍어내는 코드를 만들지 않습니다. 대신 객체 지향적인 '전략 패턴(Strategy Pattern)'을 사용하여, BehaviorStrategy라는 기본 인터페이스를 만들고 그 아래에 Image_MO_Behavior, Stat_Flush_Behavior 등을 모듈화하여 개발하게 됩니다.

## 9. AXI-3 프로토콜 및 NoC 연동 특화 속성 (Advanced AXI-3 Modeling)

트래픽 제너레이터가 생성하는 최종 `Transaction` 객체와 NoC 출력 로그에는 AXI-3 버스의 물리적 특성을 반영한 다음 속성들이 반드시 포함되어야 합니다.

### 9.1. AxID 및 Write Data Interleaving
- **요구사항:** Multi-Plane(예: Y, U, V)이나 서로 다른 내부 FIFO 채널에서 발생하는 트랜잭션은 고유한 `AxID` (Transaction ID)를 가져야 합니다.
- **출력 반영:** 트래픽 로그에 `AxID={id}` 필드를 추가하여, NoC가 Out-of-order 및 Write Interleaving 처리를 시뮬레이션할 수 있도록 합니다.

### 9.2. 동적 QoS (Quality of Service) 매핑
- **요구사항:** `DMA_IP_Spec`에 정의된 `VOTF`와 `Qurgent` 속성에 따라 AXI 버스의 `AxQOS` (4-bit) 값을 동적으로 할당해야 합니다.
- **로직:**
  - `VOTF: true`인 IP는 기본적으로 가장 높은 QoS(예: 15)를 유지합니다.
  - `Qurgent: true`인 IP는 평소에는 기본 QoS(예: 8)를 내보내다가, `Behavior_Profile` 상 내부 FIFO가 고갈/포화되는 위험 상태(Watermark 도달)가 되면 일시적으로 QoS를 높여(예: 12) 트랜잭션을 발생시킵니다.
- **출력 반영:** 트래픽 로그에 `qos={value}` 필드 추가.

### 9.3. Partial Transfer 및 WSTRB (Write Strobe)
- **요구사항:** 이미지의 Stride나 Width가 64Byte로 나누어떨어지지 않아, Line의 마지막 트랜잭션이 64Byte 경계를 다 채우지 못하는 경우를 정확히 계산해야 합니다.
- **로직:** Transaction 객체에 `valid_bytes` 속성을 추가하고, 64B 컨테이너 안에 실제 유효한 데이터가 몇 Byte인지 기록합니다.
- **출력 반영:** `bytes=64 valid_bytes=56` 형식으로 출력하여 NoC 팀이 Partial Write/Read를 인지할 수 있게 합니다.

### 9.4. AxCACHE 및 SBWC Partial Allocation
- **요구사항:** 일반 Raster 트랜잭션과 SBWC(압축 포맷) 트랜잭션의 AXI 캐시 속성을 구분해야 합니다.
- **로직:** SBWC 관련 트랜잭션(Header/Payload)이 발생할 때는 Transaction 객체의 `AxCACHE` 속성을 `Partial_Allocation_Cacheable` 상태로 태깅합니다.
- **출력 반영:** 로그에 `cache=Normal` 또는 `cache=SBWC_Alloc` 형태로 명시합니다.
